# WheelGuard Firmware

STM32F401-based WheelGuard firmware generated from STM32CubeMX and updated to preserve the legacy application behavior for button control, dual-motor PWM drive, INA219 battery/current monitoring, LED indications, and UART debug reporting.

## Migration Summary

Date: 2026-04-23

This migration ports the older hand-written WheelGuard application logic into the new CubeMX-generated project while keeping the generated peripheral handles, clocking, GPIO labels, and initialization flow as the source of truth.

The migrated logic preserves:
- Startup initialization, PWM startup, and forced motor-off boot behavior
- IDLE, CHECK_BATTERY, RUNNING, SUCCESS_BLINK, and FAULT state behavior
- Low-battery, overload, and stall safety handling
- Polled button debounce with rising-edge event generation
- INA219 configuration, retry-on-failure behavior, and moving-average filtering
- UART status output every 500 ms
- Battery-health LED behavior and success/fault blink patterns

## CubeMX Logical Mapping

| Logical Role | New CubeMX Mapping |
| --- | --- |
| Button input | `Push_Button_Pin` on `Push_Button_GPIO_Port` = `PB4`, input, active-high with internal `GPIO_PULLDOWN` |
| Red LED | `RED_LED_Pin` on `RED_LED_GPIO_Port` = `PA0` |
| Green LED | `GREEN_LED_Pin` on `GREEN_LED_GPIO_Port` = `PA2` |
| Blue LED | `BLUE_LED_Pin` on `BLUE_LED_GPIO_Port` = `PA3` |
| Motor 1 PWM | `htim3`, `TIM_CHANNEL_1`, output on `PA6` |
| Motor 2 PWM | `htim2`, `TIM_CHANNEL_1`, output on `PA5` |
| INA219 | `hi2c1` / `I2C1`, `PB6` SCL and `PB7` SDA |
| Debug UART | `huart1` / `USART1`, `PA9` TX and `PA10` RX |

## Notes On CubeMX Adaptation

- All persistent custom application logic was placed inside `/* USER CODE BEGIN ... */` and `/* USER CODE END ... */` blocks in `Core/Src/main.c`.
- No generated code outside `USER CODE` blocks was modified.
- `TIM2` and `TIM3` were generated by CubeMX as output-compare timing channels. They are reconfigured inside `USER CODE BEGIN TIMx_Init 2` sections to `PWM1` so the generated timer instances and channels can be used for the legacy motor behavior without editing CubeMX-owned code.
- Motor compare values are scaled against the generated auto-reload register so the legacy 500/900 compare targets preserve effective duty ratio if the CubeMX PWM period changes on a future regeneration.

## Preserved Legacy Behavior

- Motors start in PWM mode and are immediately forced OFF at boot.
- The application starts in `IDLE`.
- INA219 initialization is attempted once at startup and retried later from the periodic sensor update path if it fails.
- Battery qualification happens before any run.
- Motors remain OFF during IDLE, battery check, success indication, and all faults.
- Low battery blocks run start and returns to IDLE after the timed low-battery indication.
- Overload and stall faults require a new button press and return to `CHECK_BATTERY`, not directly to `RUNNING`.
- All threshold decisions use filtered moving averages.
- UART output remains human-readable and reports either filtered voltage/current data or an INA219 read error.

## Assumptions

- The push button is active-high on `PB4` and should use an internal pulldown.
- The generated timer period remains usable for the requested motor duty ratios.
- UART remains debug-only and no receive-side protocol behavior is required.

## Unresolved Issues

- None currently identified in the migrated firmware logic. Future CubeMX regenerations should preserve the button pulldown because the `.ioc` now declares it and `main.c` also reapplies it in a `USER CODE` block.
